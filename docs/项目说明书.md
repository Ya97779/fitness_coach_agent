# FitCoach AI - 智能健身与营养顾问系统

## 项目概述

FitCoach AI 是一个基于多 Agent 架构的智能健身与营养顾问系统。系统通过 LangGraph 编排多个专业 Agent（闲聊助手、营养师、健身教练、专家评审），结合 RAG 检索增强生成技术，为用户提供个性化的健身指导和营养建议。

### 核心特性

- **多 Agent 协作**：4 个专业 Agent 各司其职，通过智能路由分发用户请求
- **RAG 检索增强**：支持混合检索、HyDE、Self-RAG、CoT 等高级检索策略
- **记忆系统**：用户画像、对话历史摘要、统计数据汇总，实现个性化服务
- **专家评审**：Agent 输出经过专家评分，低分自动重试（最多 3 次）
- **前后端分离**：FastAPI 后端 + Streamlit 前端，流式响应支持

## 核心模块详解

### 1. Agent 系统

#### 1.1 路由机制

路由 Agent 使用 LLM 分析用户输入，将请求分发到合适的 Agent：

| 路由目标 | 触发条件 | 处理方式 |
|---------|---------|---------|
| Chat | 日常问候、情感交流、通用知识 | 直接生成回复，无工具调用 |
| Nutrition | 食物热量、饮食计划、营养问题 | 工具调用 + 专家评审 |
| Fitness | 运动训练、动作指导、健身计划 | 工具调用 + 专家评审 |

#### 1.2 Agent 工具集

**营养师 Agent 工具：**
- `get_user_nutrition_info` - 获取用户营养相关信息
- `log_food_intake` - 记录食物摄入
- `get_daily_nutrition_summary` - 获取每日营养摘要
- `search_food_nutrition` - 查询食物营养信息（调用天行数据 API）

**健身教练 Agent 工具：**
- `get_user_fitness_info` - 获取用户健身相关信息
- `log_exercise` - 记录运动消耗
- `estimate_exercise_calories` - 估算运动热量（MET 值计算）
- `search_fitness_knowledge` - 搜索健身知识（RAG 检索）

#### 1.3 专家评审机制

专家 Agent 对营养师和健身教练的输出进行 1-5 分评分：

| 评分 | 处理方式 |
|-----|---------|
| ≥ 3 | 通过，返回给用户 |
| < 3 | 自动重试，最多 3 次 |

评分维度：专业性(30%)、个性化(25%)、实用性(25%)、安全性(20%)

---

### 2. RAG 检索系统

#### 2.1 检索模式

| 模式 | 说明 | 适用场景 |
|-----|------|---------|
| Hybrid | 向量检索 + BM25 关键词检索 | 默认模式，平衡语义和关键词匹配 |
| Vector | 纯向量检索 | 语义相似度搜索 |
| BM25 | 纯关键词检索 | 精确关键词匹配 |
| Query Expansion | 多查询扩展 | 复杂问题，需要多角度检索 |
| HyDE | 假设性文档嵌入 | 先生成假设答案，再检索相似文档 |

#### 2.2 高级功能

- **Self-RAG**：自我反思纠正，评估检索结果质量
- **CoT 思维链**：分步推理，提高复杂问题的回答质量
- **Agentic RAG**：由 LLM 自主决定检索策略
- **增量索引**：启动时自动检测新文档并增量添加

#### 2.3 文档处理流程

```
原始文档 → 高级处理(表格/代码检测) → 智能分割 → 预处理(去重) → 向量存储
```

---

### 3. 记忆系统

#### 3.1 组件架构

```
MemoryManager
├── UserProfileLoader      # 用户画像（身高/体重/BMR/TDEE/目标）
├── ConversationSummarizer # 对话历史摘要（超过10条自动压缩）
└── StatsSummarizer        # 统计汇总（每日/每周摄入/消耗）
```

#### 3.2 记忆注入流程

1. 加载用户画像（基本信息、身体指标、目标）
2. 获取当日统计（摄入/消耗/热量缺口）
3. 获取本周统计趋势（仅营养师和健身教练）
4. 检测对话长度，必要时进行摘要压缩
5. 将所有信息注入到 System Prompt

### 4. 数据模型

```sql
-- 用户表
User {
    id: Integer (PK)
    height: Float          -- 身高(cm)
    weight: Float          -- 体重(kg)

FoodItem {
    id: Integer (PK)
    log_id: Integer (FK → DailyLog)
    name: String           -- 食物名称
    calories: Float        -- 热量

-- 运动记录表
ExerciseItem {
    id: Integer (PK)
    log_id: Integer (FK → DailyLog)
    type: String           -- 运动类型
    duration: Integer      -- 时长(分钟)
    calories: Float        -- 消耗热量
}

-- 对话日志表
ConversationLog {
    id: Integer (PK)
    user_id: Integer (FK → User)
    session_id: String     -- 会话ID
    agent_type: String     -- Agent类型
    user_message: String   -- 用户消息
    agent_response: String -- Agent回复
    created_at: DateTime
}
```

## 改进建议

### 1. 架构层面

**问题：Agent 路由可靠性不足**
- 当前路由使用 LLM 判断数字，容易误判
- 建议：引入关键词预筛选 + LLM 二次确认的混合路由策略

**问题：专家评审增加延迟**
- 每次专业回答都要经过评审，增加 1-2 次 LLM 调用
- 建议：对高频简单问题（如"苹果热量"）设置快速通道，跳过评审

**问题：流式响应实现粗糙**
- 当前是逐字符 sleep 0.01s 模拟流式，非真正 SSE
- 建议：使用真正的 token 级流式输出，或改用 WebSocket

### 2. RAG 系统

**问题：知识库内容单一**
- 目前只有 1 本 PDF，知识覆盖不足
- 建议：扩充知识库，添加运动解剖学、营养学教材、常见问题 FAQ

**问题：检索质量难以评估**
- 缺少检索结果的置信度阈值
- 建议：添加检索质量评分，低于阈值时提示用户问题不够具体

**问题：向量库未持久化更新**
- 增量索引只在启动时执行，运行时添加的知识无法及时生效
- 建议：添加知识库热更新 API

### 3. 记忆系统

**问题：对话摘要质量不稳定**
- 摘要依赖 LLM，质量受模型能力影响
- 建议：添加摘要质量校验，或使用更结构化的摘要模板

**问题：用户画像更新机制缺失**
- 用户信息创建后无法自动更新（如体重变化）
- 建议：添加周期性提醒，或支持手动更新触发重新计算

**问题：跨会话记忆断裂**
- 当前只加载最近 7 天对话，长期记忆丢失
- 建议：引入长期记忆存储，提取用户偏好和习惯

### 4. 数据安全

**问题：缺少用户认证**
- 所有 API 无鉴权，任何人可访问任何用户数据
- 建议：添加 JWT 认证或 Session 管理

**问题：输入校验不足**
- 部分 API 缺少参数校验（如负数体重、超大年龄）
- 建议：使用 Pydantic 模型添加完整校验

**问题：SQL 注入风险**
- 虽然使用 SQLAlchemy ORM，但部分原生查询需检查
- 建议：全面审查数据库查询，确保参数化

### 5. 性能优化

**问题：每次请求都创建 LLM 实例**
- Agent 函数内每次调用都 `ChatOpenAI()`，浪费资源
- 建议：使用单例或连接池管理 LLM 实例

**问题：数据库连接未优化**
- 每次工具调用都 `SessionLocal()`，缺少连接池配置
- 建议：配置 SQLAlchemy 连接池，或使用 FastAPI 依赖注入

**问题：RAG 检索未缓存**
- 相同查询每次都重新检索
- 建议：添加查询结果缓存（Redis 或内存 LRU）

## 总结

FitCoach AI 是一个架构设计合理的智能健身顾问系统，多 Agent 协作和 RAG 检索是其核心亮点。当前版本已完成基础功能闭环，但在生产化方面仍有较大提升空间。

**优先级建议：**

1. **高优先级**：添加用户认证、完善错误处理、优化流式响应
2. **中优先级**：扩充知识库、添加缓存机制、完善测试覆盖
3. **低优先级**：多模态支持、分布式架构、移动端开发

项目具备良好的扩展性基础，LangGraph 的工作流设计使得新增 Agent 或调整流程成本较低，RAG 系统的模块化设计也便于后续升级检索策略。
