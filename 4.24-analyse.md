# FitCoach AI 项目结构与代码分析文档

**分析日期：2026-04-24**

## 一、项目概述

FitCoach AI 是一个基于大语言模型（LLM）和多 Agent 架构的智能私人营养师与健身教练应用。项目采用 **LangGraph** 框架实现状态机驱动的多 Agent 协作，并集成了现代化的 **RAG（检索增强生成）** 系统获取专业知识。

### 技术栈

| 层级 | 技术选型 |
|------|---------|
| 后端框架 | FastAPI + Uvicorn |
| Agent 框架 | LangGraph |
| 数据库 | SQLite + SQLAlchemy ORM |
| 前端 | Streamlit |
| LLM 调用 | OpenAI API (兼容格式) |
| 向量数据库 | ChromaDB |
| 依赖管理 | Python 3.10+ |

## 三、后端核心模块分析

### 3.1 FastAPI 主入口 (main.py)

**文件路径**：`backend/app/main.py`

**核心功能**：
- 用户管理 API（创建用户、获取用户信息）
- 日常日志管理 API（记录摄入/消耗热量）
- 聊天接口 `/chat` 和流式聊天接口 `/chat/stream`
- 健康指标计算（BMR、TDEE）

**关键 API 端点**：

| 端点 | 方法 | 功能 |
|------|------|------|
| `/user/` | POST | 创建新用户（自动计算 BMR、TDEE） |
| `/user/{user_id}` | GET | 获取用户信息 |
| `/user/{user_id}/logs` | GET | 获取用户日志 |
| `/user/{user_id}/today` | GET | 获取当日日志 |
| `/chat` | POST | 聊天接口（非流式） |
| `/chat/stream` | POST | 流式聊天接口 |

**BMR/TDEE 计算公式**：
```python
# 男性：BMR = 10 * 体重(kg) + 6.25 * 身高(cm) - 5 * 年龄 + 5
# 女性：BMR = 10 * 体重(kg) + 6.25 * 身高(cm) - 5 * 年龄 - 161
# TDEE = BMR * 1.375（久坐办公人群）
```

### 3.3 Agents 模块详解

#### 3.3.1 基础配置 (base.py)

**文件路径**：`backend/app/agents/base.py`

**核心组件**：

1. **AgentConfig 数据类**：定义 Agent 的名称、角色、描述、系统提示词、可用工具列表

2. **AgentResponse 数据类**：定义 Agent 返回结果，包含代理名称、响应内容、是否需要评审、评审反馈

3. **MultiAgentState 类**：LangGraph 多 Agent 系统状态定义
   ```python
   {
       messages: List[BaseMessage],      # 消息历史
       user_id: int,                      # 用户 ID
       user_profile: Dict,                # 用户档案（身高、体重等）
       daily_stats: Dict,                 # 当日统计（摄入/消耗热量）
       current_agent: str                  # 当前处理的 Agent
   }
   ```

4. **AGENT_SYSTEM_PROMPTS 字典**：存储四个 Agent 的专业化 System Prompt
   - `chat`: 闲聊助手 - 日常对话、情感支持
   - `nutrition`: 营养师 - 饮食计划、热量计算
   - `fitness`: 健身教练 - 训练计划、动作指导
   - `expert`: 专家评审 - 评估输出质量（专业性30%/个性化25%/实用性25%/安全性20%）

#### 3.3.2 闲聊助手 (chat_agent.py)

**职责**：处理日常对话、问候、情感支持，不涉及专业建议

**特点**：
- 无需工具调用，直接回复
- 友好亲切，使用 emoji 增加亲和力
- 当用户询问专业问题时，引导至对应专业 Agent

#### 3.3.3 营养师 Agent (nutrition_agent.py)

**职责**：饮食规划、热量计算、营养建议

**可用工具**：
| 工具名 | 功能 |
|--------|------|
| `get_user_nutrition_info` | 获取用户营养相关信息 |
| `log_food_intake` | 记录用户摄入的食物 |
| `get_daily_nutrition_summary` | 获取当日营养摄入总结 |
| `search_food_nutrition` | 从 API 查询食物营养信息 |

**工作流程**：
```
用户询问食物问题 → LLM 判断是否需要工具 → 执行 search_food_nutrition 
→ 获取 API 结果 → LLM 整合生成优化回答 → 返回用户
```

**API 集成**：调用天行数据食物营养 API，支持 100+ 种食物查询

#### 3.3.4 健身教练 Agent (fitness_agent.py)

**职责**：训练计划、动作指导、运动科学

**可用工具**：
| 工具名 | 功能 |
|--------|------|
| `get_user_fitness_info` | 获取用户健身相关信息 |
| `log_exercise` | 记录用户运动及消耗 |
| `estimate_exercise_calories` | 使用 MET 值估算运动消耗 |
| `search_fitness_knowledge` | 从 RAG 知识库检索专业知识 |

**MET 值参考**：
| 运动类型 | 轻度 | 中度 | 高强度 |
|----------|------|------|--------|
| 跑步 | 7 | 10 | 14 |
| 游泳 | 6 | 10 | 14 |
| HIIT | 8 | 12 | 17 |
| 力量训练 | 4 | 6 | 8 |

**RAG 集成**：使用 ModernRAG 从健身知识库检索专业内容

#### 3.3.5 专家评审 Agent (expert_agent.py)

**职责**：评估营养师和健身教练的输出质量

**评分维度**：
- 专业性（30%）：内容是否科学准确
- 个性化（25%）：是否考虑用户个人情况
- 实用性（25%）：是否易于执行
- 安全性（20%）：是否存在潜在风险

**评分标准**：
| 评分 | 含义 | 动作 |
|------|------|------|
| 1-2 分 | 严重不足 | 需要重试 |
| 3 分 | 基本合格 | 通过 |
| 4-5 分 | 优秀 | 通过 |

**评审流程**：
```
提取评分 → 判断是否 ≥ 3 → 通过则结束 / 不通过则返回 Agent 重试（最多 3 次）
```

#### 3.3.6 路由 Agent (router.py)

**职责**：分析用户意图，决定路由到哪个 Agent

**路由判断逻辑**：
```
包含"饮食/热量/营养" → nutrition Agent
包含"运动/训练/健身" → fitness Agent  
日常闲聊/问候/通用问题 → chat Agent
```

**实现方式**：使用 LLM 进行意图识别，返回 1/2/3 数字代表对应 Agent

#### 3.3.7 LangGraph 工作流编排 (graph.py)

**文件路径**：`backend/app/agents/graph.py`

**整体架构**：
```
START → router → [chat | nutrition → expert_review | fitness → expert_review] → END
```

**节点定义**：
| 节点 | 功能 | 是否支持工具调用 |
|------|------|----------------|
| `router` | 意图识别，路由分发 | 否 |
| `chat` | 闲聊助手 | 否 |
| `nutrition` | 营养师 | 是（search_food_nutrition 等） |
| `fitness` | 健身教练 | 是（search_fitness_knowledge 等） |
| `expert_review` | 专家评审 | 否 |

**边路由规则**：
```
router → chat (闲聊) / nutrition / fitness
chat → END (直接结束)
nutrition → expert_review → [nutrition 重试 | END]
fitness → expert_review → [fitness 重试 | END]
expert_review → 重试条件：评分 < 3 且重试次数 < 3
```

**状态重试机制**：最多重试 3 次，防止无限循环

**核心函数**：
- `create_llm()`: 创建 LLM 实例（支持自定义 temperature）
- `router()`: 路由节点
- `chat()`: 闲聊节点
- `nutrition()`: 营养师节点
- `fitness()`: 健身教练节点
- `expert_review()`: 专家评审节点
- `process_user_message()`: 对外暴露的主入口函数

## 四、RAG 系统分析

### 4.1 系统架构概览

**文件路径**：`backend/app/rag/__init__.py`

ModernRAG 是一个模块化的现代 RAG 系统，支持多种检索和生成增强策略。

**支持的功能**：
| 类别 | 功能 | 说明 |
|------|------|------|
| **基础检索** | 向量检索 | ChromaDB 语义检索 |
| **基础检索** | BM25 | 关键词精确匹配 |
| **混合检索** | Hybrid Search | 向量 + BM25 + RRF 融合 |
| **查询增强** | Query Expansion | 多查询扩展 |
| **查询增强** | HyDE | 假设性文档嵌入 |
| **生成增强** | CoT | 思维链推理 |
| **生成增强** | Self-RAG | 自我反思纠正 |
| **自主决策** | Agentic RAG | LLM 自主选择策略 |

### 4.2 核心模块详解

#### 4.2.1 BM25 关键词检索 (bm25.py)

BM25（Best Matching 25）是一种经典的文本检索算法，用于关键词精确匹配。

**核心参数**：
- `k1`: 控制词频饱和度（默认 1.5）
- `b`: 控制文档长度归一化（默认 0.75）

**优点**：
- 精确匹配关键词
- 不依赖语义理解
- 计算效率高

#### 4.2.2 混合检索 (hybrid_search.py)

结合向量检索和 BM25 的优势，使用 **RRF（Reciprocal Rank Fusion）** 算法融合分数。

**RRF 公式**：
```python
RRF_score = Σ 1/(k + rank_i)
# k: 融合参数（通常 60）
# rank_i: 某检索方法的结果排名
```

**权重配置**：
```python
vector_weight: float = 0.5  # 向量检索权重
bm25_weight: float = 0.5    # BM25 权重
```

#### 4.2.3 Query Expansion (query_expansion.py)

将用户的一个查询扩展为多个不同角度的查询，提升检索覆盖率。

**示例**：
```
原始查询："深蹲怎么做"
↓ 扩展
["深蹲的正确姿势", "深蹲动作要领", "新手如何学习深蹲", "深蹲常见错误"]
```

#### 4.2.4 HyDE 假设性文档嵌入 (hyde.py)

**核心思想**：
1. 让 LLM 生成一个"假设性答案"
2. 用这个假设性答案去检索相关文档
3. 用检索到的真实文档回答用户问题

**优点**：当用户查询表述与知识库文档差异较大时效果显著

#### 4.2.5 CoT 思维链推理 (cot.py)

**核心思想**：在生成最终答案前，先生成推理步骤

**流程**：
```
问题 → 推理步骤1 → 推理步骤2 → ... → 最终答案
```

**适用场景**：分析性、推理性问题

#### 4.2.6 Self-RAG 自我反思 (self_rag.py)

**核心思想**：让 LLM 评估检索结果和回答质量，自我反思纠正

**评估标准**：
1. 准确性：是否有事实错误
2. 完整性：是否完整回答问题
3. 相关性：是否紧密围绕问题
4. 清晰度：表达是否清晰

#### 4.2.7 Agentic RAG 自主决策 (agentic_rag.py)

**核心思想**：让 LLM 作为大脑，自主决定检索和生成策略

**Router Agent 决策树**：
```
用户问题 → Router Agent 分析
    ├── need_retrieval: true/false
    ├── retrieval_strategy: no_retrieval/basic/hyde/query_expansion/cot/self_rag
    ├── generation_strategy: direct/rag_based/cot/self_reflect
    └── suggested_top_k: 3-5
```

**策略类型**：
| 策略 | 适用场景 |
|------|----------|
| `no_retrieval` | 闲聊、观点讨论 |
| `basic` | 一般事实性问题 |
| `hyde` | 查询与文档表述差异大 |
| `query_expansion` | 复杂问题、多角度问题 |
| `cot` | 分析性、推理性问题 |
| `self_rag` | 重要问题、高质量要求 |

### 4.3 文档处理流程

```
原始文档 → Loader（加载）→ Preprocessor（清洗）→ Splitter（分割）→ 索引
                                                          ↓
用户查询 ←─────────────── ←─────────────── ←──────────── Retrieval（检索）
                    ↓
              Generator（生成） → 最终回答
```

**处理模块**：
| 模块 | 功能 |
|------|------|
| `DocumentLoader` | 支持 PDF、DOCX、TXT 等多格式加载，带重试机制 |
| `TextPreprocessor` | 文本清洗（去除噪声、标准化） |
| `IntelligentSplitter` | 智能分割（语义分块、保留表格/代码块） |
| `TableDetector` | 表格检测与提取 |
| `CodeBlockDetector` | 代码块检测与保留 |
| `SemanticChunker` | 语义分块 |

## 九、改进建议（基于代码分析）

### 9.1 架构层面

1. **Agent 工具调用循环限制**：当前 `nutrition_with_user` 和 `fitness_with_user` 中工具调用循环最多 2 次，建议可配置化

2. **RAG 实例管理**：`_rag_instance` 使用全局变量懒加载，可考虑使用依赖注入提升可测试性

3. **错误处理**：部分模块缺少完整的异常捕获，可增强错误提示

### 9.2 功能层面

1. **流式输出优化**：Expert Review 节点结果目前不支持流式，建议增强

2. **缓存机制**：RAG 检索结果、API 查询结果可增加缓存层

3. **用户画像**：可增加更多用户属性（运动偏好、饮食限制、时间安排等）

### 9.3 安全性

1. **输入验证**：建议增加更严格的输入长度和内容过滤

2. **API Key 管理**：敏感信息建议使用密钥管理服务

3. **医疗免责声明**：当前已有边界说明，可考虑更显著的位置展示

## 十、总结

FitCoach AI 是一个架构清晰、技术选型合理的 AI Agent 项目：

**优点**：
- 多 Agent 协作架构设计合理，职责划分清晰

- 数据库设计简洁实用，满足核心业务需求

- 添加完整的单元测试
- 考虑微服务架构拆分
- 增加监控和日志系统
