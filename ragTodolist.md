## 🔍 现有 RAG 系统问题分析

### 1. 文档处理问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **文档加载不稳定** | 使用 `PyPDFLoader`、`Docx2txtLoader`，部分文档加载失败无重试机制 | 知识库不完整 |
| **分割策略简单** | 固定 `chunk_size=500, chunk_overlap=50`，未考虑文档结构 | 语义连贯性差 |
| **缺少文本清洗** | 直接加载原始文本，未做标准化处理 | 检索噪声多 |
| **多模态支持弱** | 图片使用 `UnstructuredImageLoader`，OCR 效果有限 | 图片信息丢失 |

### 2. 向量检索问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **只用纯语义检索** | `similarity_search` 无混合检索 | 同义词/多义词检索效果差 |
| **缺少重排序** | 直接返回 top-k 结果，未做 `rerank` | 相关性排序不准确 |
| **向量化模型单一** | 统一用 `embedding-2`，未针对不同类型文档优化 | 检索精度受限 |
| **无查询改写** | 用户 query 直接检索，未做 `query expansion` | 口语化 query 效果差 |

### 3. 检索增强问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **上下文窗口利用不足** | 简单拼接 top-3 文档，未做上下文压缩 | 超出 LLM token 限制 |
| **缺少引用追溯** | 返回结果无 `citation` 信息 | 用户无法核实来源 |
| **未做知识去重** | 相似内容重复检索 | 回答冗余 |
| **缺少对话历史利用** | 每次检索独立，未结合历史 context | 多轮对话效果差 |

### 4. 系统架构问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **单例 vectorstore** | 全局初始化，不支持动态更新 | 热更新困难 |
| **无增量更新** | 新增文档需全量重建 | 更新成本高 |
| **缺少索引管理** | 无版本控制、无分区管理 | 知识库混乱 |
| **错误处理简陋** | `try-except` 简单捕获，无降级策略 | 稳定性差 |

### 5. 可观测性问题

| 问题 | 现状 | 影响 |
|------|------|------|
| **无检索日志** | 不记录 query、召回结果、用户反馈 | 优化无依据 |
| **无质量评估** | 缺少 `Hit Rate`、`MRR` 等指标 | 效果难量化 |
| **无 A/B 测试支持** | 无法对比不同检索策略 | 迭代效率低 |

## 🎯 现代 RAG 系统架构目标

### 目标特性

```
┌─────────────────────────────────────────────────────────────┐
│                     Modern RAG Pipeline                       │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐ │
│  │  Query   │───▶│  Query  │───▶│  Retrieve│───▶│  Rerank │ │
│  │ Preproc  │    │ Expansion│   │          │    │         │ │
│  └──────────┘    └──────────┘    └──────────┘    └────┬────┘ │
│                                                      │      │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐         │      │
│  │ Context  │◀───│  LLM     │◀───│  Graded  │◀────────┘      │
│  │ Compress │    │ Generate │    │ Results  │                  │
│  └──────────┘    └──────────┘    └──────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

### 核心能力

- **多跳推理**：支持 `Multi-Hop` 检索（如"这个动作的常见错误和如何避免？"）
- **混合检索**：`Dense（向量）+ Sparse（关键词）` 混合
- **上下文压缩**：智能裁剪无关上下文，节省 token
- **可追溯引用**：每个回答附带来源citation
- **动态知识库**：支持增量更新，在线切换

## 📋 RAG 升级任务清单

### 第一阶段：基础增强（低风险，快速见效）

- [x] **1.1** 改进文档加载器，添加重试机制和错误日志
- [x] **1.2** 实现智能文本分割器（考虑段落、标题、表格结构）
- [x] **1.3** 添加文本预处理（去重、标准化、特殊字符处理）
- [x] **1.4** 增加 BM25 关键词检索作为补充
- [x] **1.5** 实现 `Hybrid Search`（向量 + BM25 融合）
- [x] **1.6** 实现高级文档处理器（语义分割、表格提取、代码块保留）
- [x] **1.7** 实现文档结构分析（标题层级、章节树）
- [x] **1.8** 实现上下文感知清洗（保留格式、列表结构）

### 第二阶段：检索优化（中等风险）

- [x] **2.1** 接入 `Cohere Rerank` 或本地 `bge-reranker` 重排序
- [x] **2.2** 实现查询改写（Query Expansion / HyDE）
- [ ] **2.3** 添加查询分类（闲聊 / 事实性 / 分析性）分流处理
- [ ] **2.4** 实现上下文窗口压缩（LLM-based Context Compression）
- [ ] **2.5** 支持知识去重（MinHash / SimHash）

### 第三阶段：高级特性（高风险高回报）

- [x] **3.1** 实现 `Self-RAG` 或 `Corrective-RAG` 自我纠正
- [ ] **3.2** 支持多跳推理（Iterative Retrieval + Graph）
- [ ] **3.3** 实现流式输出（Streaming RAG）
- [ ] **3.4** 添加对话历史感知的检索（Conversation History）
- [x] **3.5** 实现 `Agentic RAG`（LLM 自主决定检索策略）

### 第四阶段：生产化（稳定性保障）

- [ ] **4.1** 动态知识库管理（增量更新、版本控制）
- [ ] **4.2** 检索质量监控（Hit Rate、MRR、Latency）
- [ ] **4.3** 实现 A/B 测试框架
- [ ] **4.4** 用户反馈收集与微调数据积累

## 🔧 技术选型建议

### 检索增强组件

| 组件 | 推荐方案 | 备注 |
|------|----------|------|
| **向量数据库** | ChromaDB（当前）/ Qdrant / Milvus | 考虑切换到 Qdrant（性能更好） |
| **Embedding** | embedding-3（当前）/ BGE / Jina | 考虑多语言 BGE |
| **Reranker** | Cohere / BGE-reranker-base | 本地部署更安全 |
| **分割器** | LangChain + 自定义 / LlamaIndex | 自定义更灵活 |
| **混合检索** | RRF fusion / Weighted fusion | 融合向量和 BM25 |

### 评估工具

| 指标 | 工具 | 说明 |
|------|------|------|
| **Hit Rate** | 自建日志 | Top-K 召回率 |
| **MRR** | 自建日志 | 平均倒数排名 |
| **RAGAS** | ragas 库 | 现代 RAG 评估框架 |
| **Trulens** | trulens-eval | 全链路追踪 |

## ⚠️ 风险与注意事项

1. **API 成本**：Rerank、Embedding 调用会增加 API 消耗
2. **延迟**：多阶段检索会增加响应时间，需做流式输出
3. **准确性 vs 召回**：过度压缩可能丢失关键信息
4. **冷启动问题**：新知识库缺少评估数据，需人工标注
